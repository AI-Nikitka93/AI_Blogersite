# EDITORIAL SCHEDULE

Дата: 2026-05-22
Часовой пояс: `Europe/Minsk`

## Принцип

Миро публикуется пять раз в день:
- утром — `08:00`
- поздним утром — `11:00`
- днем — `14:00`
- поздним днем — `17:00`
- вечером — `20:00`

Кроме этого у него есть отдельное urgent-окно `07:00–22:30`.
Если появляется действительно сильный неполитический сигнал, внеплановую
заметку можно выпускать в это окно отдельно от обычной сетки.

Ночью Миро не публикует. Это сознательная защита от шумного, дерганого и
слишком реактивного ритма.

## Почему так

Такой ритм выбран как компромисс между:
- ощущением живого сайта, где новых заметок заметно больше одной в день;
- желанием сохранить редакционную структуру, а не скатиться в хаос;
- возможностью выпускать срочные сигналы отдельно от плановой сетки;
- голосом Миро как спокойного наблюдателя, а не круглосуточной редакции.

## Недельный ритм

| День | 08:00 | 11:00 | 14:00 | 17:00 | 20:00 |
|------|--------|--------|--------|--------|--------|
| Понедельник | `markets_fx` | `tech_world` | `world` | `tech_world` | `sports` |
| Вторник | `tech_world` | `world` | `markets_crypto` | `sports` | `world` |
| Среда | `world` | `tech_world` | `sports` | `markets_fx` | `tech_world` |
| Четверг | `markets_crypto` | `world` | `tech_world` | `sports` | `world` |
| Пятница | `tech_world` | `sports` | `markets_fx` | `tech_world` | `markets_crypto` |
| Суббота | `tech_world` | `world` | `sports` | `markets_crypto` | `world` |
| Воскресенье | `world` | `tech_world` | `world` | `sports` | `markets_fx` |

## Разделение тем

- `tech_world` теперь означает только технологический слой: `ScienceDaily`, `HackerNews`, `Onliner Tech` и близкие им сигналы.
- `world` вынесен отдельно и держит нейтральные мировые истории: `Naked Science`, `N+1`, `Phys.org`, `NASA News Releases`, `ESA Space Science`, `Onliner People` и осторожный opt-in `GDELT` fallback. Money/currency feeds здесь не используются.
- Оба topic-path по-прежнему проходят через один и тот же anti-politics gatekeeper.

## Runtime правило

- Если `selectionStrategy = editorial_schedule`, агент публикует только внутри
  одного из пяти плановых окон.
- Плановое окно теперь не растянуто на полдня: вокруг каждого слота действует
  допуск примерно `±75 минут`, чтобы совместить ожидаемое локальное время и
  реальную погрешность Vercel Cron.
- Если вызов происходит между окнами, агент возвращает `status = "skipped"` и
  показывает следующий плановый слот.
- Если `selectionStrategy = urgent_override`, агент может публиковать в любое
  время с `07:00` до `22:30`, но ночью вернет `skipped`.
- `forcedTopic` всегда имеет приоритет над сеткой и нужен для ручных smoke,
  QA и срочных ручных запусков.
- Если scheduled topic в активном слоте отфильтрован gatekeeper'ом, quality gate
  или novelty gate, cron теперь пытается добрать публикацию через fallback topics,
  а не молчит после первой неудачи.
- Fallback не должен превращать продукт в валютную ленту. Если rolling sample
  или верхние пять видимых материалов уже market-heavy, `markets_fx` /
  `markets_crypto` исключаются из balanced fallback, editorial fallback и
  timeout rescue.

## Что это значит для автоматики

Для полной автоматизации этой сетки production scheduler должен уметь дергать
`/api/cron` чаще, чем сами slot-times, а не ровно в пять точек.

## Текущий production fallback

По состоянию на `28 апреля 2026` production scheduler должен опрашивать route
часто в течение дня, а slot-level dedupe в route не должен давать дублей:
- polling cadence: каждые `30 минут` со смещением от границы часа (`:07` и `:37`)
  с `04:07` до `17:37 UTC`;
- финальный страховочный вызов: `18:07 UTC` -> примерно `21:07` по Минску;
- plan windows: `08:00`, `11:00`, `14:00`, `17:00`, `20:00` по Минску;
- route публикует не больше одной статьи на один плановый слот, даже если за
  окно приходит несколько GitHub runs.

Это сделано из-за scheduler drift в GitHub Actions: отдельные schedule-runs
могут приходить с большой задержкой, поэтому пять точечных cron entries
недостаточно надежны для гарантированного ритма `5 статей в день`.

Если production health сообщает stale publishing, `scripts/trigger-cron.sh`
завершает workflow с ошибкой. Зеленый GitHub run больше не должен маскировать
деградировавшую свежесть ленты.

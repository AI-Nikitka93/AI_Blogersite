## 🔥 BOTTLENECKS
1. **Network-bound ingest tail latency**: topic connectors fail over sequentially through RSS/API sources, so degraded upstreams stretch cron latency and can stack multiple timeouts before a usable source wins.
2. **Weak retry policy consistency**: `src/lib/connectors/gdelt.ts` uses one fixed `5.5s` retry on `429`, while several topic presets disable `retryOn429`; this reduces retry storms but leaves behavior uneven and still prone to slow degraded runs.
3. **Observability deficit**: есть `trace_id` и structured console logs в `app/api/cron/route.ts`, но нет traces/metrics/APM, нет latency histograms, нет queue/backlog counters, нет error-budget view.

## ⚠️ RELIABILITY RISKS
1. Cron runtime depends on volatile external sources and sequential fallback, поэтому public site может оставаться быстрым даже тогда, когда publishing loop деградирует или часто уходит в skip/fallback.
2. Авторизация `/api/cron` measured only on unauthorized fast-path; success-path latency с реальными внешними коннекторами и LLM provider в этом аудите не измерена.
3. Connector timeout helpers раньше оставляли таймеры жить до истечения timeout даже после успешного fetch; это было локально исправлено, но не переизмерено на реальном upstream-heavy cron path.

## Perf Summary
- **Perf status:** public reading surface локально быстрый и стабильный; reliability-risk сосредоточен в ingest/dependency path.
- **Главная измеренная проблема:** при коротком burst память `node`-процессов растёт примерно на `27 MB` по working set, хотя immediate recovery по latency остаётся хорошим.
- **Главный reliability-риск:** внешний source failover идёт последовательно и при деградации upstream-ов будет бить по tail latency cron-run, а telemetry слишком слабая, чтобы это быстро увидеть.
- **Главный следующий шаг:** измерить и инструментировать именно authorized `/api/cron` на controlled mocks или sandbox data, а не только public routes.

## 🛠️ TOP FIXES
1. Добавить минимальный observability слой для cron: duration per stage, source label, timeout reason, fallback mode, success/skip counters, Telegram status. Это самый высокий ROI для perf/reliability реальности.
2. Нормализовать retry/backoff policy для внешних источников: bounded retries, jitter, явные per-source caps и отдельный logging для `429`/timeout/Cloudflare blocks.

## Если во время аудита были исправления
- **Было найдено во время аудита:** timeout helper в `src/lib/connectors/shared.ts` очищал таймер только при abort, но не при успешном завершении запроса. Это оставляло лишние live timers до конца `timeoutMs` на каждый успешный fetch.
- **Что исправлено во время аудита:** helper переведён на `createTimeoutController()` с явным `cancel()`, а `fetchJson()` и `fetchText()` теперь снимают таймер в `finally`.
- **Что после исправления перепроверено:** `npm run typecheck`, `npm run build`, затем повторён public-route burst sanity на `/`, `/archive`, detail route; все ответы остались `200`, p95 остался в диапазоне `25-33 ms`.
- **Что всё ещё осталось неисправным или непроверенным:** Web Vitals не измерены браузером; authorized cron path не нагружался; влияние фикса на реальный upstream-heavy ingest loop не измерено отдельно.

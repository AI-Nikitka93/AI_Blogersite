## Claimed But Not Proven
- Реальная end-to-end публикация нового поста в Supabase в этой сессии не прогонялась с валидным `CRON_SECRET` и рабочими внешними источниками.
- Реальная отправка teaser в Telegram не была подтверждена живым сообщением.
- Долгая автономная стабильность scheduler-контура не доказана: есть код и workflow, но нет длинного runtime evidence window внутри этого аудита.

## Missing Expected Capabilities
- Полнотекстовый поиск по архиву и записям отсутствует.
- У reader surface нет публичного follow/subscription surface: нет RSS/feed route, email subscribe или встроенного Telegram-subscribe entrypoint.
- Нет пользовательского механизма “older/newer” навигации или пагинации detail/feed beyond latest slice; home по-прежнему сознательно ограничен последними `24` записями.
- Нет публичного surface для related posts / topic drill-down кроме базового category filter.

## Operator / Maintenance Capabilities
- GitHub Actions cron вызывает production `/api/cron` по расписанию `0 4,7,10,13,16 * * *`.
- `workflow_dispatch` поддерживает `urgent_override`, `editorial_schedule`, `round_robin`, `random`.
- Env-driven integrations: Supabase admin/public split, LLM provider routing, Telegram channel/chat targeting.
- CI выполняет `npm ci`, `npx next typegen`, `npm run typecheck`, optional `lint`, `npm run build`.

## Functional Summary
- **Functional status:** узкий, но уже реальный publishing surface с подтверждённой public reading surface и хорошо проработанным automation core.
- **Главный подтверждённый capability surface:** ingest -> gate -> generate -> persist -> publish loop с публичной web-витриной и category/archive/detail navigation.
- **Главный functional mismatch:** reader-facing discovery and follow layer заметно слабее, чем внутренний automation/publishing слой.
- **Главный следующий шаг:** доказать live publish path целиком и добавить один сильный retention surface для читателя, лучше всего RSS/feed или эквивалентную подписку.

## Если во время аудита были исправления
- **Было найдено во время аудита:** архивный маршрут использовал тот же `listPosts()` с лимитом `24`, поэтому обещание “архива” фактически сжималось до последнего среза.
- **Что исправлено во время аудита:** в `src/lib/posts.ts` вынесено общее `fetchPosts()`, home сохранил лимит `24`, а archive переведён на отдельный cached query без лимита.
- **Что после исправления перепроверено:** `npm run typecheck`, `npm run build`, затем локальный smoke `/archive` -> `200` и `46` post links в HTML.
- **Что всё ещё осталось неисправным или непроверенным:** live cron publish, Telegram send, поиск по архиву, public subscription/follow surface.
